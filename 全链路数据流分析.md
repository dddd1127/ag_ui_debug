> 本文档追踪从用户输入到 Agent 推理再回到 UI 展示的完整数据流，

## 二、请求阶段：UI → Agent（全链路追踪）

### 2.1 用户输入 → AG-UI 请求

**文件**: `web/src/hooks/useAguiSSE.ts:119-148`

```typescript
function buildAguiRequest(text: string, threadId: string, history: ChatMessage[]): RunAgentInput {
  const messages = history.map(msg => ({
    id: msg.id,
    role: msg.role,
    content: msg.content,     // ← 注意：content 是累积全量字符串
  }))
  messages.push({ id: `msg_${Date.now()}`, role: 'user', content: text })
  return { threadId, runId, messages, tools: [], context: [], ... }
}
```

**问题 P1 — 历史消息全量累积**：
- `history` 中每条消息的 `content` 是**全量累积文本**（由 SSE 流逐步 `+=` 拼接而来）
- 每轮请求都会把所有历史消息全量发送给后端，这是 AG-UI 协议的正常行为
- 但在多轮长对话中，请求体会持续膨胀

### 2.2 AG-UI 请求 → agentscope Msg 列表

**文件**: `server/agent_runner.py:159-267` (`convert_agui_messages_to_msgs`)

```
AG-UI Message.role="user"    → Msg(name="user", role="user", content=[TextBlock(text=全量)])
AG-UI Message.role="system"  → Msg(name="system", role="system", content=[TextBlock(text=全量)])
AG-UI Message.role="assistant" → Msg(name="Friday", role="assistant", content=[TextBlock + ToolUseBlock...])
AG-UI Message.role="tool"    → Msg(name="tool", role="assistant", content=[ToolResultBlock(output=全量)])
```

**这一阶段无增量问题**：输入侧本就应该是完整消息，AG-UI 协议要求历史消息全量传入。

### 2.3 调试推送：请求转换

**文件**: `server/agui_endpoint.py:187-204`

```python
field_mappings = [
    {"from": "threadId", "to": "thread_id", "value": thread_id},
    {"from": "runId", "to": "run_id", "value": run_id},
    {"from": "messages", "to": "messages", "value": f"{len(run_agent_input.messages)} 条消息"},
]
```

推送 `direction="request"` 的 `agui_transform` 事件到 WebSocket。
**这一阶段无问题**，仅做字段映射展示。

## 三、推理阶段：Agent 内部处理（全量 → 增量的关键转换）

### 3.1 stream_printing_messages 返回全量 Msg

**文件**: `server/agent_runner.py:301-304`

```python
async for raw_msg, last in stream_printing_messages(
    agents=[agent],
    coroutine_task=agent(msgs),
):
```

**⚠️ 核心问题来源**：`stream_printing_messages` 是 agentscope 1.x 的流式输出接口，
每次 yield 的 `raw_msg` 中 TextBlock.text 是**累积全量文本**，不是增量 delta。

举例（3 次 yield）：

| yield 次序 | raw_msg.content TextBlock.text | last |
|---|---|---|
| 第1次 | `"你"` | False |
| 第2次 | `"你好"` | False |
| 第3次 | `"你好，我"` | True |

### 3.2 _compute_delta_msg：全量 → 增量

**文件**: `server/agent_runner.py:319-405`

```python
def _compute_delta_msg(raw_msg, msg_id, is_last, sent_text_len_map):
    # 对于 TextBlock：
    full_text = block.get("text", "")
    sent_len = sent_text_len_map.get(msg_id, 0)
    delta = full_text[sent_len:]        # ← 截取增量
    sent_text_len_map[msg_id] = len(full_text)

    if delta or is_last:                 # ← 关键判断
        new_blocks.append(TextBlock(type="text", text=delta))
    # 如果 delta 为空且 is_last=False → 不添加任何 block
```

**转换后示例**：

| yield 次序 | delta_msg.content | is_last |
|---|---|---|
| 第1次 | `[TextBlock(text="你")]` | False |
| 第2次 | `[TextBlock(text="好")]` | False |
| 第3次 | `[TextBlock(text="，我")]` | True |

### 3.3 ⚠️ 增量计算的已知问题

#### 问题 P2 — 空 delta 导致 Msg(content=[])

**触发条件**：`stream_printing_messages` 两次 yield 之间文本内容未变化（模型还在"思考"但没产出新 token）

```
sent_text_len_map[msg_id] = 5   # 已发 5 字符
raw_msg TextBlock.text = "你好"  # 仍为 5 字符（未变）
delta = "你好"[5:] = ""          # 空 delta
is_last = False
→ if delta or is_last: False    # 不添加 TextBlock
→ Msg(content=[])               # 空 content
```

**后续影响链**：

```
Msg(content=[])
  → msg_to_agui_events(): content_blocks = [], for 循环不执行, events = []
    → agui_endpoint.py: agui_events 为空
      → response_field_mappings.append({"from": "msg_type", "to": "无", "value": "Msg 无 AG-UI 对应事件"})
      → _extract_input_summary() 返回 {"type": "none", "summary": "无对应 AG-UI 事件"}
        → 调试面板显示 "❌ 无对应 AG-UI 事件"
```

#### 问题 P3 — invocation_id 缺失导致增量计算错误

**文件**: `server/agent_runner.py:308`

```python
msg_id = getattr(raw_msg, "invocation_id", None) or getattr(raw_msg, "id", "")
```

如果 agentscope 1.x 的某些 Msg 没有 `invocation_id`，则 `msg_id` 退化为空字符串 `""`。
所有消息共享同一个 key `""`，导致 `sent_text_len_map[""]` 的值被不同消息互相覆盖，
**增量计算完全错误**——可能出现 delta 为负索引、跳字、或重复输出。

#### 问题 P4 — ReActAgent 多轮推理的增量追踪

ReActAgent 的推理循环：

```
轮次1: 模型输出文本 "让我查一下..." + ToolUseBlock(get_weather)
轮次2: ToolResultBlock 返回天气数据
轮次3: 模型输出文本 "北京今天晴天..."
```

每轮推理模型返回的 `raw_msg` 具有不同的 `invocation_id`。但 `sent_text_len_map`
以 `msg_id` 为 key，**不同轮次使用不同的 msg_id，所以增量追踪天然隔离，不会跨轮串扰**。

但有一个边界情况：如果同一轮推理中，模型先返回纯文本 chunk，再返回 ToolUseBlock，
此时 `is_last` 的语义是"这一条流式消息的最后一块"。如果 agentscope 在 ToolUseBlock
出现的 chunk 上标记 `is_last=True`，则文本消息会提前发送 `TEXT_MESSAGE_END`，
后续如果还有文本 chunk，`tracker._current_message_id` 已被清空，会创建新的 message_id，
导致前端显示为两条独立消息。

## 四、协议转换阶段：Msg → AG-UI 事件

### 4.1 msg_to_agui_events 核心逻辑

**文件**: `server/agui_adapter.py:83-155`

```
输入: Msg(content=[TextBlock, ToolUseBlock, ToolResultBlock, ...])
输出: List[AGUIEvent]

转换规则：
┌─────────────────────────┬──────────────────────────────────────────┐
│ Msg content block       │ 产生的 AG-UI 事件序列                     │
├─────────────────────────┼──────────────────────────────────────────┤
│ TextBlock(text=delta)   │ TEXT_MESSAGE_START → CONTENT → END       │
│ ToolUseBlock            │ TEXT_MESSAGE_END(关闭文本) →              │
│                         │ TOOL_CALL_START → ARGS → END             │
│ ToolResultBlock         │ TEXT_MESSAGE_END(关闭文本) →              │
│                         │ TOOL_CALL_RESULT                         │
│ str 内容                │ TEXT_MESSAGE_START → CONTENT → END       │
│ 其他类型                │ ⚠️ 静默跳过                               │
│ Msg(content=[])         │ ⚠️ 返回空列表 → "无对应 AG-UI 事件"       │
└─────────────────────────┴──────────────────────────────────────────┘
```

### 4.2 ⚠️ 未识别 block type 的问题

**文件**: `server/agui_adapter.py:118-154`

```python
for block in content_blocks:
    block_type = block.get('type') if isinstance(block, dict) else None

    if block_type == 'text':         # ✅ 处理
        ...
    elif block_type == 'tool_use':   # ✅ 处理
        ...
    elif block_type == 'tool_result':# ✅ 处理
        ...
    elif isinstance(block, str):     # ✅ 处理
        ...
    # ❌ 其他 block_type 全部静默跳过
```

**可能出现的未识别 block type**：

| block_type | 来源 | 说明 |
|---|---|---|
| `thinking` | 支持 CoT 的模型 | DeepSeek/通义千问等模型的思考链内容 |
| `image` | 多模态模型 | 图片内容块 |
| `refusal` | 安全过滤 | 模型拒绝响应的标记 |
| `redacted_thinking` | Claude 模型 | 加密思考链 |
| `input_json_delta` | 某些模型 | 工具调用参数的增量块 |
| 自定义类型 | agentscope 扩展 | 框架未来可能新增的类型 |

同时，`_compute_delta_msg` 对未知类型做了**透传**（`agent_runner.py:396-398`），
但 `msg_to_agui_events` 不认识它们 → **所有未知 block 被静默丢弃** → 空事件列表 → "无对应"。

### 4.3 AGUIEventTracker 状态管理

**文件**: `server/agui_adapter.py:50-81`

```
tracker 维护的状态：
- _run_started: bool          → 确保 RUN_STARTED 只发一次
- _run_finished: bool         → 确保 RUN_FINISHED 只发一次
- _text_started: set[msg_id]  → 哪些 message_id 已发 TEXT_MESSAGE_START
- _text_completed: set[msg_id]→ 哪些 message_id 已发 TEXT_MESSAGE_END
- _current_message_id: str    → 当前流式输出的 message_id
- _current_tool_call_id: str  → 当前工具调用的 tool_call_id
```

**问题 P5 — TEXT_MESSAGE_START/END 配对风险**：

当 `Msg(content=[])` 导致 `msg_to_agui_events` 返回空列表时，tracker 的状态**不会更新**。
如果前一个 chunk 已经创建了 `_current_message_id` 并发了 `TEXT_MESSAGE_START`，
但后续 chunk 因为空 delta 被跳过，则 `TEXT_MESSAGE_END` 一直不会发出，
直到 `is_last=True` 的 chunk 到来才关闭。这不会导致前端崩溃，但会导致调试面板中
该消息的"结束"事件与实际内容结束时间不一致。

## 五、调试推送阶段：双重数据通道

### 5.1 两条独立的数据通道

```
通道1: SSE 流 → 前端 ChatPanel（用户可见的聊天界面）
通道2: WebSocket → 前端 DebugPanel（开发者调试面板）
```

两条通道的数据**来源于同一个处理流程**，但推送的内容和格式不同：

| 维度 | SSE 通道 | WebSocket 通道 |
|---|---|---|
| 协议 | HTTP SSE | WebSocket |
| 方向 | 服务端 → 客户端单向 | 服务端 → 客户端单向 |
| 内容 | AG-UI 标准事件 | 调试专用 JSON |
| 格式 | `data: {json}\n\n` | `{"type": "...", "data": {...}}` |
| 用途 | ChatPanel 渲染对话 | DebugPanel 展示转换细节 |

### 5.2 调试推送的时机和内容

**文件**: `server/agui_endpoint.py:225-280`

```python
async for msg, is_last in run_agent_stream(agent, msgs):
    # 5a. Msg → AG-UI 事件
    agui_events = msg_to_agui_events(msg, is_last, tracker)

    # 5b. 推送调试：Agent 内部事件（基于 AG-UI 事件，非原始 Msg）
    for agui_event in agui_events:
        debug_mapping = _map_agui_to_debug_event(agui_event)
        if debug_mapping:              # ← 跳过 RUN_STARTED/FINISHED/TEXT_MESSAGE_START/TOOL_CALL_END
            await publisher.publish_agent_event(...)

    # 5c. 推送调试：响应转换
    if agui_events:
        response_field_mappings = [{"from": "msg_type", "to": "type", "value": "..."}]
    else:
        response_field_mappings = [{"from": "msg_type", "to": "无", "value": "Msg 无 AG-UI 对应事件"}]

    input_summary = _extract_input_summary(agui_events)  # ← {"type": "none"} 或增量摘要
    await publisher.publish_response_transform(
        agent_event=input_summary,    # ← 输入（增量摘要）
        agui_events=agui_events_data, # ← 输出（AG-UI 事件列表）
        field_mappings=response_field_mappings,
    )

    # 5d. 序列化 SSE
    for agui_event in agui_events:
        yield serialize_agui_event(agui_event)
```

### 5.3 ⚠️ 调试推送与 SSE 流的一致性问题

**问题 P6 — 调试推送可能多出 "无对应" 条目**：

当 `Msg(content=[])` 时：
- SSE 通道：不输出任何事件（正确，空消息无需输出）
- 调试通道：**仍然推送**一条 `response_transform`，标记为 "无对应 AG-UI 事件"

这导致调试面板中出现红色 ❌ 标记，但实际上这只是空 chunk 的正常过滤结果，
**并非协议转换的真正错误**。用户看到 ❌ 容易误解为出了问题。

**问题 P7 — 调试 input 与 SSE output 的内容语义差异**：

- 调试的 `input` 使用 `_extract_input_summary()` 提取的增量摘要
- SSE 的 output 是完整的 AG-UI 事件

这两个都是增量语义，**是一致的**。但如果 `_extract_input_summary` 遇到未知事件类型，
它只提取 `TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_ARGS`、`TOOL_CALL_RESULT`
四种类型的数据，其他类型（如 `THINKING_TEXT_MESSAGE_CONTENT`）的信息会丢失。

## 六、响应阶段：SSE 事件 → 前端 ChatPanel

### 6.1 SSE 解析与状态更新

**文件**: `web/src/hooks/useAguiSSE.ts:256-353`

```typescript
for (const line of lines) {
  const event = parseSSELine(trimmedLine)  // 解析 data: {json}

  setMessages(prev => {
    switch (event.type) {
      case 'TEXT_MESSAGE_CONTENT':
        assistant.content += event.delta || ''   // ← 增量追加
        break
      case 'TOOL_CALL_START':
        assistant.toolCalls.push({ id, name, arguments: '', status: 'calling' })
        break
      case 'TOOL_CALL_ARGS':
        lastTool.arguments += event.delta || ''  // ← 增量追加
        break
      case 'TOOL_CALL_RESULT':
        lastTool.result = event.content || ''
        break
      case 'RUN_FINISHED':
        assistant.status = 'completed'
        break
    }
  })
}
```

**问题 P8 — content 全量累积在前端**：

`assistant.content += event.delta` 逐步将增量 delta 拼接成全量字符串。
这是前端渲染的正常需要（需要显示完整文本），但：

1. `buildAguiRequest` 会把 `history` 中的全量 `content` 再次发送给后端
2. 长对话中 `content` 会越来越长，但这是 AG-UI 协议设计使然

**问题 P9 — 多轮 ToolCall 追踪可能错位**：

```typescript
case 'TOOL_CALL_RESULT':
  if (assistant.toolCalls && assistant.toolCalls.length > 0) {
    const lastTool = assistant.toolCalls[assistant.toolCalls.length - 1]
    lastTool.result = event.content || ''
  }
```

如果 ReActAgent 在一次推理中调用多个工具，AG-UI 事件序列为：

```
TOOL_CALL_START(id=A, name=get_weather)
TOOL_CALL_ARGS(id=A, delta=...)
TOOL_CALL_END(id=A)
TOOL_CALL_START(id=B, name=get_time)
TOOL_CALL_ARGS(id=B, delta=...)
TOOL_CALL_END(id=B)
TOOL_CALL_RESULT(id=A, ...)     ← 注意：结果可能乱序
TOOL_CALL_RESULT(id=B, ...)
```

当前代码用 `toolCalls[toolCalls.length - 1]` 取最后一个工具，
但 `TOOL_CALL_RESULT` 的顺序可能与 `TOOL_CALL_START` 不一致（先返回 B 的结果再返回 A 的结果），
导致**结果赋值给错误的工具调用**。

**问题 P10 — 单条助手消息绑定所有事件**：

当前实现中，一次 `sendMessage` 创建**一个** `assistantMsg`，所有 AG-UI 事件
（包括多轮推理的文本 + 工具调用 + 再推理的文本）都追加到同一条消息的 `content` 和 `toolCalls` 中。

但在 ReActAgent 场景中，Agent 可能执行多轮推理：
```
轮次1: "让我查一下天气" + tool_call(get_weather)
轮次2: tool_result(天气数据)
轮次3: "北京今天晴天，温度25°C"
```

理想情况下，前端应显示为**一条消息**（包含文本+工具调用卡片+后续文本），
但当前 `content += delta` 的拼接方式会把所有文本拼成一个字符串，
工具调用穿插在中间，导致**显示顺序与实际推理顺序不一致**。

### 7.2 Agent 内部事件面板 AgentInfoView

**文件**: `web/src/components/DebugPanel/AgentInfoView.tsx`

将 Agent 事件组织为5个步骤：
1. 接收消息 (ReplyStartEvent)
2. 思考过程 (ThinkingBlock*Event)
3. 生成回复 (TextBlock*Event)
4. 工具调用 (ToolCall*Event + ToolResult*Event)

**静默丢失**。

## 十三、总结

1. **"无对应 AG-UI 事件" 的主要根因**是 `stream_printing_messages` 两次 yield 之间文本未变化，
   导致 `_compute_delta_msg` 产出空 Msg，`msg_to_agui_events` 返回空事件列表。

2. **次要根因**是 `msg_to_agui_events` 只识别 3 种 block type（text/tool_use/tool_result），
   不支持 thinking 等新类型，导致数据丢失后也出现 "无对应"。

3. **全量问题**主要存在于两个层面：
   - **源头**：`stream_printing_messages` 天然返回全量累积文本，由 `_compute_delta_msg` 负责切割为增量
   - **前端**：`content += delta` 将增量拼接回全量（渲染需要），并在下一轮请求中全量发送回后端（协议要求）

4. **最应优先修复的三个问题**：
   - **P2**：空 Msg 导致调试面板误报 "无对应" → 建议跳过空事件的推送
   - **P9**：ToolCall 结果按"最后一个"匹配 → 改为按 toolCallId 匹配
   - **P3**：invocation_id 缺失导致增量计算错误 → 增加 fallback 机制

# 第二部分：RUN_FINISHED 丢失根因分析

> AG-UI 协议要求每个 Run 必须以 RUN_FINISHED 收尾。
> 但在实际运行中，前端经常收不到 RUN_FINISHED 事件，
> 导致助手消息永远停留在 `streaming` 状态。

---

## 十四、RUN_FINISHED 的发送路径

### 当前代码中 RUN_FINISHED 的唯二产出点

```
generate_agui_stream() 的两条代码路径：

路径 A（正常结束）：agui_endpoint.py:303-306
─────────────────────────────────────────
    async for msg, is_last in run_agent_stream(...):
        ...                              ← 循环体
                                       ← 循环自然结束，进入 ↓
    # 7. 确保 RUN_FINISHED
    finish_events = build_run_finished_event(tracker)
    for event in finish_events:
        yield serialize_agui_event(event)   ← ✅ RUN_FINISHED 被发出

路径 B（异常结束）：agui_endpoint.py:308-327
─────────────────────────────────────────
    except Exception as e:
        error_events = build_run_error_event(tracker, ...)
        for event in error_events:
            yield serialize_agui_event(event)   ← RUN_ERROR
        finish_events = build_run_finished_event(tracker)
        for event in finish_events:
            yield serialize_agui_event(event)   ← ✅ RUN_FINISHED 被发出
```

**乍看之下，两条路径都能产出 RUN_FINISHED。但实际运行中仍然丢失，为什么？**

---

## 十五、RUN_FINISHED 丢失的五大根因

### 根因 R1：异步生成器被提前放弃，没有 finally 块 [最关键]

**这是最核心的设计缺陷。**

`generate_agui_stream` 是一个 `async generator`（使用 `yield`），
它的清理代码（产出 RUN_FINISHED 的代码）**不在 `finally` 块中**，
而是在 `try` 块的尾部。

```python
async def generate_agui_stream(...) -> AsyncGenerator[str, None]:
    try:
        async for msg, is_last in run_agent_stream(agent, msgs):
            yield serialize_agui_event(...)    # ← 产出 SSE 数据
            # ... 调试推送 ...
            yield serialize_agui_event(...)

        # ★★★ 以下代码只在生成器被完整消费后才执行 ★★★
        finish_events = build_run_finished_event(tracker)
        for event in finish_events:
            yield serialize_agui_event(event)  # ← RUN_FINISHED

    except Exception as e:
        # ★★★ 以下代码只在 try 内抛异常时才执行 ★★★
        finish_events = build_run_finished_event(tracker)
        for event in finish_events:
            yield serialize_agui_event(event)  # ← RUN_FINISHED
```

**当生成器没有被完整消费时，两条路径都不执行！**

具体触发场景：

```
┌──────────────────────────────────────────────────────────────────┐
│                      generate_agui_stream 生成器                  │
│                                                                  │
│  yield RUN_STARTED                                               │
│  yield TEXT_MESSAGE_CONTENT(delta="你")                          │
│  yield TEXT_MESSAGE_CONTENT(delta="好")                          │
│  yield TEXT_MESSAGE_CONTENT(delta="，我")                         │
│  yield TEXT_MESSAGE_END                                          │
│                                                                  │
│  ← 此时客户端断开连接 / 超时 / 页面刷新                           │
│  ← FastAPI/Starlette 调用生成器的 aclose()                       │
│  ← 生成器收到 GeneratorExit 信号                                  │
│                                                                  │
│  ★ 以下代码永远不会执行 ★                                        │
│  finish_events = build_run_finished_event(tracker)               │
│  yield serialize_agui_event(RunFinishedEvent)                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**为什么不能加 `finally`？**

Python 语法**禁止在 `finally` 块中使用 `yield`**：

```python
try:
    yield data
finally:
    yield final_data   # ← RuntimeError: generator ignored GeneratorExit
```

这是 Python 语言层面的限制（PEP 533 讨论了这个问题，但至今未解决）。

---

### 根因 R2：`stream_printing_messages` 无超时，可能永远挂起

**文件**: `server/agent_runner.py:301-304`

```python
async for raw_msg, last in stream_printing_messages(
    agents=[agent],
    coroutine_task=agent(msgs),
):
```

如果底层模型 API 无响应（网络问题、API 限流、服务端卡住），
`stream_printing_messages` 会**永远等待**，`async for` 循环永远不会结束。

此时：
- 生成器卡在 `await` 上
- 无法到达 `build_run_finished_event`
- 前端永远收不到 RUN_FINISHED
- 助手消息永远显示 "接收中..."

```
时间线：
t=0s    前端发送请求
t=0.1s  收到 RUN_STARTED
t=0.2s  收到 TEXT_MESSAGE_CONTENT(delta="你")
t=0.3s  stream_printing_messages 等待下一个 token...
t=30s   仍在等待...
t=60s   仍在等待...（API 无响应）
        ← 永远不会有 RUN_FINISHED
```

---

### 根因 R3：ReActAgent 多轮推理中 `is_last` 的语义歧义

**文件**: `server/agent_runner.py:301-314`

```python
async for raw_msg, last in stream_printing_messages(
    agents=[agent],
    coroutine_task=agent(msgs),
):
    msg_id = getattr(raw_msg, "invocation_id", None) or getattr(raw_msg, "id", "")
    delta_msg = _compute_delta_msg(raw_msg, msg_id, last, sent_text_len_map)
    yield delta_msg, last
```

`stream_printing_messages` 的 `last` 参数含义是 **"这是否是整个流式输出的最后一条消息"**。

但在 ReActAgent 的推理循环中，一次推理包含多轮：

```
轮次1: 模型输出 "让我查一下天气" + ToolUseBlock(get_weather)
        → last=False? last=True? ← 取决于 agentscope 内部实现

轮次2: ToolResultBlock 返回天气数据
        → last=False?

轮次3: 模型输出 "北京今天晴天，温度25°C"
        → last=True
```

**如果 agentscope 在轮次1结束时就标记 `last=True`**：

```
轮次1: yield Msg("让我查一下天气" + ToolUse), last=True
       ↓ _convert_text_block: is_last=True → 发出 TEXT_MESSAGE_END
       ↓ tracker._current_message_id = ""   ← 被清空

轮次2: yield Msg(ToolResult), last=False
       ↓ _convert_tool_result_block: 正常
       
轮次3: yield Msg("北京今天晴天"), last=False
       ↓ tracker._current_message_id = "" → 创建新的 message_id
       ↓ 发出新的 TEXT_MESSAGE_START → 前端看到两条独立消息
```

更严重的是：**如果 `last=True` 在轮次1就出现，而 `stream_printing_messages` 在轮次3之后才真正结束循环**，
那么 `async for` 循环在轮次3之后才自然退出，RUN_FINISHED 才会发出。
但如果 `last=True` 在轮次1出现后，下游代码可能误以为推理已结束，
而实际上还有更多消息要来——这是一个**语义不一致**问题。

**如果 agentscope 在整个推理完成后才标记 `last=True`（只有最后一条消息）**：
这是正确行为，但需要确认 agentscope 1.x 的实际实现。

### 根因 R4：RUN_STARTED 从未发出 → RUN_FINISHED 变成孤例

**文件**: `server/agui_adapter.py:105-113`

```python

    events = []
    
    # RUN_STARTED 只在第一次调用 msg_to_agui_events 时发出
    if not tracker._run_started:
        tracker._run_started = True
        events.append(RunStartedEvent(...))
    
    # 处理 content blocks...
```

**RUN_STARTED 的发送条件**：`msg_to_agui_events` 至少被调用一次。

如果 `run_agent_stream` 产出零条消息（`stream_printing_messages` 直接结束，无任何输出）：

```
async for msg, is_last in run_agent_stream(agent, msgs):
    # ← 循环体从未执行
    pass

# 6. ReplyEndEvent（调试推送，正常执行）
# 7. RUN_FINISHED

# tracker._run_started = False  ← RUN_STARTED 从未发出
# 但 build_run_finished_event 不检查 _run_started，直接发 RUN_FINISHED
```

结果：SSE 流中只包含 `RUN_FINISHED`，没有 `RUN_STARTED`。

这违反 AG-UI 协议的事件序列约束：
```
正确序列：RUN_STARTED → ... → RUN_FINISHED
错误序列：RUN_FINISHED                 ← 协议违规
```

某些 AG-UI 客户端可能会**忽略没有前置 RUN_STARTED 的 RUN_FINISHED**，
导致前端认为 Run 仍在运行。

### 根因 R5：前端 SSE 解析的 buffer 截断

**文件**: `web/src/hooks/useAguiSSE.ts:256-267`

```typescript
while (true) {
    const { done, value } = await reader.read()
    if (done) break                   // ← 流结束时退出循环
    
    buffer += decoder.decode(value, { stream: true })
    const lines = buffer.split('\n')
    buffer = lines.pop() || ''         // ← 保留不完整的最后一行
    
    for (const line of lines) {
        // 处理完整的 SSE 行
        const event = parseSSELine(trimmedLine)
        // ...
    }
}
```

**问题**：当 SSE 流关闭时（`done=true`），`buffer` 中可能还有**未处理的残余数据**：

```
网络传输的分包可能导致：

chunk1: "data: {\"type\":\"RUN_FINISHED\""
chunk2: ",\"threadId\":\"thread_xxx\"}\n\n"

→ 第一轮 read: buffer = "data: {\"type\":\"RUN_FINISHED\""
  split('\n') → lines = ["data: {\"type\":\"RUN_FINISHED\""]
  → 这不是完整的 SSE 行，parseSSELine 解析失败
  → buffer 保留残余

→ 第二轮 read: done=true → 退出循环
  ★ buffer 中的残余数据 "data: {\"type\":\"RUN_FINISHED\"..." 永远不会被处理 ★
```

这是**最常见的 RUN_FINISHED 丢失原因**：事件确实被服务端发出了，
但在最后一个 TCP 包中，SSE 数据跨包分割，前端读取时 `done=true` 直接退出，
**没有处理 buffer 中的残余数据**。

## 十七、各根因的发生概率与影响

| 根因 | 发生概率 | 影响严重度 | 触发条件 |
|---|---|---|---|
| **R1: 无 finally 块** | 高 | 致命 | 客户端断开、页面刷新、网络中断 |
| **R2: 无超时机制** | 中 | 致命 | API 无响应、网络卡住 |
| **R3: is_last 语义歧义** | 低 | 中 | ReActAgent 多轮推理（取决于 agentscope 实现） |
| **R4: RUN_STARTED 缺失** | 低 | 中 | Agent 无输出（空回复） |
| **R5: buffer 截断** | 高 | 严重 | 最后一个 SSE 事件跨 TCP 包 |

**R1 和 R5 的叠加效应**是最常见的场景：
客户端在流式传输中途刷新页面 → 生成器被放弃 → RUN_FINISHED 不发出；
即使不刷新，最后的 RUN_FINISHED 事件也可能因为 TCP 分包被前端 buffer 截断。

```typescript
function processSSEEvent(event: AGUIEvent, assistantMsgId: string, setMessages: Function) {
    setMessages(prev => {
        const newMessages = [...prev]
        const assistantIndex = newMessages.findIndex(m => m.id === assistantMsgId)
        if (assistantIndex === -1) return prev
        const assistant = { ...newMessages[assistantIndex] }
        
        switch (event.type) {
            case 'TEXT_MESSAGE_CONTENT':
                assistant.content += event.delta || ''
                break
            case 'RUN_FINISHED':
                assistant.status = 'completed'
                break
            // ... 其他 case
        }
        
        newMessages[assistantIndex] = assistant
        return newMessages
    })
}

// 主循环中使用
while (true) {
    const { done, value } = await reader.read()
    if (done) {
        // ★ 处理残余 buffer
        if (buffer.trim()) {
            const event = parseSSELine(buffer.trim())
            if (event) processSSEEvent(event, assistantMsgId, setMessages)
        }
        break
    }
    buffer += decoder.decode(value, { stream: true })
    const lines = buffer.split('\n')
    buffer = lines.pop() || ''
    for (const line of lines) {
        const event = parseSSELine(line.trim())
        if (event) processSSEEvent(event, assistantMsgId, setMessages)
```

### 后果 1：助手消息永远卡在 "接收中..."

```

  R1: 客户端断开 → 生成器放弃 → RUN_FINISHED 不发出

  ┌──────────────────────────┐
```

```
C10: 如果有 TOOL_CALL_START，应该有 TOOL_CALL_RESULT

当前验证情况：
  C1: AGUIEventTracker._run_started 保证只发一次 ✅
  C2: AGUIEventTracker._run_finished 保证只发一次 ✅（但不保证一定发出 ❌）
  C3: 代码逻辑上 RUN_STARTED 在最前面 ✅（但不保证 Agent 有输出时也发出 ❌）
  C4: ❌ 无验证
  C5: AGUIEventTracker._text_started/_text_completed 保证配对 ✅
      （但不保证一定发出 END ❌，见 R1/R2）
  C6: 代码逻辑上 START/END 配对 ✅
  C7-C10: ❌ 无验证
```

### 方案 1：服务端 Run 生命周期注册表 [核心]

```python
# server/run_registry.py

import asyncio
import time
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class RunState:
    """单个 Run 的生命周期状态"""
    run_id: str
    thread_id: str
    started_at: float = field(default_factory=time.time)
    finished_at: Optional[float] = None
    
    # 事件追踪
    run_started_sent: bool = False
    run_finished_sent: bool = False
    text_messages_open: int = 0       # 已 START 但未 END 的文本消息数
    tool_calls_open: int = 0          # 已 START 但未 END 的工具调用数
    
    # 事件计数
    events_sent: int = 0
    last_event_at: Optional[float] = None
    
    # 状态
    status: str = "running"           # running / completed / error / timed_out / abandoned
    
    @property
    def duration(self) -> float:
        end = self.finished_at or time.time()
        return end - self.started_at
    
    @property
    def is_stuck(self) -> bool:
        """是否卡住（超过 N 秒无事件）"""
        if self.status != "running":
            return False
        if self.last_event_at is None:
            return time.time() - self.started_at > 30  # 启动后 30 秒无事件
        return time.time() - self.last_event_at > 60   # 60 秒无事件
    
    @property
    def is_complete(self) -> bool:
        """是否完整结束（所有配对事件都已关闭）"""
        return (
            self.run_started_sent
            and self.run_finished_sent
            and self.text_messages_open == 0
            and self.tool_calls_open == 0
        )


class RunRegistry:
    """
    全局 Run 生命周期注册表
    
    功能：
    1. 注册/注销 Run
    2. 记录事件发送情况
    3. 检测卡住的 Run
    4. 提供超时清理
    5. 提供运行状态查询 API
    """
    
    def __init__(self, default_timeout: float = 300):
        self._runs: dict[str, RunState] = {}
        self._default_timeout = default_timeout
        self._cleanup_task: Optional[asyncio.Task] = None
    
    def register(self, run_id: str, thread_id: str) -> RunState:
        state = RunState(run_id=run_id, thread_id=thread_id)
        self._runs[run_id] = state
        return state
    
    def record_event(self, run_id: str, event_type: str):
        state = self._runs.get(run_id)
        if not state:
            return
        
        state.events_sent += 1
        state.last_event_at = time.time()
        
        if event_type == "RUN_STARTED":
            state.run_started_sent = True
        elif event_type == "RUN_FINISHED":
            state.run_finished_sent = True
            state.status = "completed"
            state.finished_at = time.time()
        elif event_type == "RUN_ERROR":
            state.status = "error"
            state.finished_at = time.time()
        elif event_type == "TEXT_MESSAGE_START":
            state.text_messages_open += 1
        elif event_type == "TEXT_MESSAGE_END":
            state.text_messages_open = max(0, state.text_messages_open - 1)
        elif event_type == "TOOL_CALL_START":
            state.tool_calls_open += 1
        elif event_type == "TOOL_CALL_END":
            state.tool_calls_open = max(0, state.tool_calls_open - 1)
    
    def mark_abandoned(self, run_id: str):
        """标记 Run 为被放弃（客户端断开等）"""
        state = self._runs.get(run_id)
        if state and state.status == "running":
            state.status = "abandoned"
            state.finished_at = time.time()
    
    def get_stuck_runs(self) -> list[RunState]:
        return [s for s in self._runs.values() if s.is_stuck]
    
    def get_incomplete_runs(self) -> list[RunState]:
        """获取未正常完成的 Run（RUN_STARTED 但没有 RUN_FINISHED）"""
        return [
            s for s in self._runs.values()
            if s.run_started_sent and not s.run_finished_sent
        ]
    
    async def start_cleanup_loop(self):
        """定期清理超时的 Run"""
        while True:
            await asyncio.sleep(30)
            now = time.time()
            for state in list(self._runs.values()):
                if state.status == "running":
                    if now - state.started_at > self._default_timeout:
                        state.status = "timed_out"
                        state.finished_at = now
                        logger.warning(
                            f"Run {state.run_id} timed out after {state.duration:.1f}s, "
                            f"events_sent={state.events_sent}, "
                            f"text_messages_open={state.text_messages_open}, "
                            f"tool_calls_open={state.tool_calls_open}"
                        )
            # 清理超过 1 小时的已完成 Run
            expired = [
                rid for rid, s in self._runs.items()
                if s.finished_at and now - s.finished_at > 3600
            ]
            for rid in expired:
                del self._runs[rid]
    
    def get_status(self) -> dict:
        """获取注册表状态（供 /health 和管理 API 使用）"""
        running = [s for s in self._runs.values() if s.status == "running"]
        return {
            "total_runs": len(self._runs),
            "running": len(running),
            "stuck": len(self.get_stuck_runs()),
            "incomplete": len(self.get_incomplete_runs()),
            "runs": [
                {
                    "run_id": s.run_id,
                    "thread_id": s.thread_id,
                    "status": s.status,
                    "duration": round(s.duration, 1),
                    "events_sent": s.events_sent,
                    "text_messages_open": s.text_messages_open,
                    "tool_calls_open": s.tool_calls_open,
                    "last_event_at": s.last_event_at,
                }
                for s in running
            ],
        }


# 全局单例
registry = RunRegistry()
```

## 二十五、总结

本项目的状态检查可以用一句话概括：

> **只管"产出"，不管"送达"；只管"开始"，不管"结束"；只管"发事件"，不管"验状态"。**

具体来说：

| 维度 | 现状 | 应有 |
|---|---|---|
| 事件产出 | AGUIEventTracker 保证基本序列 | ✅ 够用 |
| 事件送达 | 无确认机制，不知道前端是否收到 | ❌ 需要 ACK 或序列号 |
| 生命周期完整 | RUN_STARTED 不保证发出，RUN_FINISHED 不保证到达 | ❌ 需要注册表 + 超时 |
| 异常检测 | 无超时、无心跳、无悬空检测 | ❌ 需要心跳 + watchdog |
| 状态恢复 | WS 重连不回补，前端刷新丢失 | ❌ 需要状态恢复机制 |
| 可观测性 | /health 只返回 "healthy" | ❌ 需要运行状态查询 |
| 自愈能力 | 前端兜底代码有 buffer 截断缺陷 | ❌ 需要超时自愈 |

**最核心的一步**：实现 RunRegistry（方案 1），让服务端知道"当前有哪些 Run 在跑、跑了多久、是否卡住"。
有了这个基础设施，超时、清理、健康检查、管理端点才能建立起来。
